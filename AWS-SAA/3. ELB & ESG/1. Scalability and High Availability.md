- Some services like databases are perfectly fine for vertical scaling.
- *Highly Available* means the service is available in 2 or more Datacenters(~AZ)
- Goal is to survive data center loss
- Types of high availability:
	- Passive:
		- One **primary instance** handles all traffic.
		- A **standby instance** sits in another Availability Zone.
		- Data is **synchronously replicated** to the standby.
		- The standby **does not serve traffic**.
		- If the primary fails, AWS performs **automatic failover** to the standby.
		- Examples
			- Amazon RDS Multi-AZ
			- Azure SQL Database active-passive failover groups
			- Traditional **database standby replicas**
		- Good for
			- Databases requiring **strong consistency**
			- Systems where **failover time of ~1 minute is acceptable**
	- Passive
		- Multiple nodes serve traffic simultaneously 
		- Traffic is **load balanced**
		- If one node fails, others **continue serving instantly** (Near Zero failover time)      
		- **Examples**
			- Amazon Aurora multi-reader cluster
			- Azure Cosmos DB multi-region writes
			- *Distributed web servers behind a **load balancer***
		- **Good for**
			- Large scale workloads
			- Systems needing **very low downtime**
			- Applications that can handle **eventual consistency**
