- Load balancers are the servers that forward requests to multiple servers.
- ELB(Elastic Load balancer) is a managed AWS service.
- It is a no-brainer to use ELB instead of creating one ourselves.(Saves time and effort)

### Types:
	- Classic Load Balancers
		- It is deprecated and old, although can be used
		- Don't use it
		- Supports: HTTP, HTTPS, TCP, TLS, SSL
	- Application Load Balancers 2016
		- Works on application layer
		- Supports: HTTP, HTTPS, Websocket
	- Network Load Balancers 2017
		- Works on layer 4 (network layer)
		- Supports: TCP, IDP, TLS
	- Gateway Load Balancer 2020
		- Also works on network layer(4) and transport layer(3)
		- firewall / security appliance insertion
		- Manages IP traffic

### Reasons to use:
	- High availability
	- Seamlessly handle failure of downstream instances 
	- Health checks for our servers 
		- to figure out if a instance is working or not, if it doesn't work, we stop forwarding requests there
		- we can configure an endpoint to enable this e.x. 'health'
	- Single point of access (DNS)
	- SSL termination, meaning SSL is decrypted at Load balancers and then forwarded to instance
		- this reduces the load on our actual EC2
		- certificates can be managed in one place
	- Separate private and public traffic
	- Stickiness with cookies

### Load Balancers and Security groups
- For load balancers:
	- Allow port 80(HTTP) inbound from 0.0.0.0/0(everywhere)
	- Allow port 443(HTTP) inbound from 0.0.0.0/0(everywhere)
	- Outbound is already enabled by default for all security groups
- For instances:
	- Allow port 80, 443 inbound ONLY from *Load Balancer's Security Group*
