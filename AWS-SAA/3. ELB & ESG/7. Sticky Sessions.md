- Generally, when we send a request to an application(which is load balanced), it goes to Machine A. Our next request to that application, may go to Machine B.
- But, applications sometimes store **session data locally on the server**.
- Sticky Sessions solve this problem. They ensure that requests from a certain user, goes to a certain machine only. So the load balancer **remembers the client**.
- But note, too much stickiness can cause the load to be imbalanced. This is why Amazon, does not prefer this approach. 
	- A better approach would be to store session data in some other store. (Redis, DynamoDB, Database)
- CLB, ALB, and NLB can have sticky sessions. CLB and ALB use cookies. Whereas, NLB, does NOT require cookies for sticky sessions.
- Cookies have an expiration, after cookie expires, user maybe routed to other instances.

## Types:
1. Application Cookies
	- Created by application
	- Custom attributes allowed
	- Unique cookie name per target group
	- Can't use AWSALB, AWSALBAPP, AWSALBTG
2. Duration based Cookies
	- Created by the load balancer
	- Most commonly used
	- Cookie Names: ALB -> AWSALB; CLB -> AWSELB;
