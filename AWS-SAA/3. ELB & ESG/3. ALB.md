### Features
- Load balancing to multiple HTTP applications across machines (target-groups)
- Load balancing to multiple applications on the same machine (containers)
- Supports redirects. (HTTP -> HTTPS)
- Useful for micro-services and container based applications.
- Has a port mapping feature, to redirect to dynamic port in ECS.
- If we used CLB, we would require 1 CLB for each Application
- Routing can be based on:
	- Path in URL (`users/` -> Instance 1, `search` -> Instance 2)
	- Hostname in URL (`maps`.google.com -> Instance 1, `mail`.google.com -> Instance 2)
	- Request headers, Query Strings (?id=20)
- ALB has a fixed hostname (xxx.region.elb.amazonaws.com)
- The Instances/Application cannot see the IP of the client directly. (Since SSL termination happened), they will have to check extra headers for that
	- `x-forwarded-for`: IP
	- `x-forwarded-port`: Port
	- `x-forwarded-proto`: Protocol
- We can add listeners, in which we provide conditions (like path or ip of the user etc) and perform action if this condition is fulfilled (ex. return 404, send request to a certain instance etc.)
### Target Groups
Each target group should have health check
ALB can route to multiple groups
1. EC2 Instances (Maybe managed by ASG)
2. ECS Tasks
3. Lambda Functions (Basis for serverless)
4. Private IP addresses (since they dont change)


- ALBs do not support attaching Elastic IP addresses directly.
- ALBs are assigned a static DNS name, but their IP addresses are dynamic and can change over time.
- Because the IPs are dynamic, they are less suitable if you need a fixed IP address for regulatory compliance or firewall rules.
