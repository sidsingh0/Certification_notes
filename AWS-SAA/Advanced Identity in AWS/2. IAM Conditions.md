Our IAM policies have allow/deny on various resources,
They also allow us to add conditions
- Conditions = **extra filters on permissions**
- They are used for:
    - IP restrictions
    - Region restrictions
    - Time-based rules (sometimes)
    - ex. **aws:SourceIp** - restrict the client IP from which the API calls are being made 
      **aws:RequestedRegion** - restrict the region the API calls are made to
      **ec2:ResourceTag** - instead of saying user can access all ec2 resources, we say access ec2 resources with following tags
      **aws:PrincipalOrgID** - You can write a policy that only allows access if the request comes from an account that belongs to your AWS Organizations.

#### IAM vs Resource Based Policies
- When we assume a IAM role, (user, application, service), we give up our original permissions and only have permissions assigned for the role.
- *In Resource Based Policies, we do not have to give up our permissions to access the resource.*
- Example: If user A scans the bucket and dumps it into a bucket in user B account.
	- Using a **resource-based policy on S3** is often easier because:
		- The user doesn’t need to switch roles
		- They can **read and write in one continuous flow**
		- It reduces complexity in cross-account access

#### Regarding Eventbridge Permissions
- When eventbridge rule runs, it requires access to the target resource
- There are 2 types of target
	1. Targets that require an IAM Role (not just “IAM policies”)
		Examples:
		- Amazon ECS (RunTask)
		- Amazon Kinesis
		- AWS Systems Manager (Run Command)
		For these:
		- EventBridge **assumes an IAM role**
		- That role has permissions like `ecs:RunTask`, `kinesis:PutRecord`, etc.
	2. The **resource itself grants permission** to EventBridge using resource level policies
		- (S3, SQS, SNS, API Gateway)
