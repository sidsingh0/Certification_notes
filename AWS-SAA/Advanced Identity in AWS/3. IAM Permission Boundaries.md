- IAM Permission boundaries define maximum attainable IAM Permissions for an entity
- It is supported for users and roles only. Groups are not supported.
- Example:
	- IAM Permission Boundary says allow on `iam:*`
	- And IAM policy for user says allow `s3:*, iam:CreateUser`
	- Then the user will be able to create users, but not access s3(since this is not mentioned in the permission boundary)
	- ![[Pasted image 20260423095547.png]]
	- Uses:
		- Used for delegation / guardrails
		- *Useful to restrict one specific user (instead of a whole account using Organizations & SCP)*
		- Allow developers to self-assign policies and manage their own permissions, while making sure they can’t “escalate” their privileges (= make themselves admin)

# IAM Identity Center
- Allows us to login to all our accounts using one sign in
	- Not just AWS accounts but also Microsoft 365, box, slack etc.
	- Custom SAML 2.0 enabled apps that we create
- No need to maintain id pw for test, dev, prod acc separately
- ![[Pasted image 20260423225415.png]]
- Here, AWS Organization has a main management account with full access SCP
- Inside the organization there are multiple OUs with accounts
- We create Users and permission sets inside AWS
- These users and permission sets permutation and combination is used by an AWS account(inside an organization)
- We assign user and permutation sets to aws account user.
- Identity center solves the following:
	- “User needs access to multiple AWS accounts without managing roles manually”
	- "SSO"
	- “Company wants to eliminate IAM users in each account”
	- Federation / external identity providers(Active Directory, Okta, Google Workspace)
- *Identity Center Supports Attribute based access control. For example, to see if bob can access prod account or not, we see his title, location, department etc.*
- Identity center also supports allowing third party applications to sign in to access aws and business apps (salesforce, Box, Office 365)

# AWS Control Tower
- It basically applies the concept of organizations, account creation, guardrails, compliance etc.
- Its job is to make the manual tasks, i.e. creating accounts manually, enviroment setup etc. easier.
- It provides a a secure landing zone, guardrails, managing many teams and accounts and create a DEFAULT SETUP
- Provides ongoing governance for your Control Tower environment (AWS Accounts) 
	- Preventive Guardrail – using SCPs (e.g., Restrict Regions across all your accounts)
	- Detective Guardrail – using AWS Config (e.g., identify untagged resources)
