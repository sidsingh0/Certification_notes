- Global service
- Manage mulitple account together as one system
- Structure
	1. Management Account
		- This is the main account
		- It creates and controls the organization
		- It also manages the billing
	2. Member Account
- Features
	1. Billing at one place (management account pays for all)
	2. Cost savings (Volume discounts, better pricing tier)
	3. Shared discounts 
		- If one account has Reserved instances or saving plans, it can be shared across all the accounts
	4. Account creation can be automated
	5. There can be OU inside OU(Organization Units)

#### Best Practices
1. Use tags for billing purposes, e.x. - `Project: PaymentApp`
2. Centralized CloudTrail logs(Who did what)
	- We can enable CloudTrail on all the accounts and send logs to a single S3 of security account
3. Centralized CloudWatch(Application logs)
4. Access other accounts with roles (instead of Id password)

#### SCPS (Security Control Policies)
- These are the policies applied to OU/Account that restrict users and roles.
- This can only be assigned by management account
- If something is not mentioned in SCP, it means it is not allowed.
	- But by default, AWS attaches "FullAWSAccess" to SCP.
	- **AllowList strategy:** So if we delete this statement, every access will be denied. -> then we can attach allow  policies to the required resource
	- **Blocklist strategy:** If we do not delete this statement, every access is allowed -> then we can attach disallow policies to restrict access to the resources 
- *Not applicable to Management Account*
- They work with IAM, to get access to a resource we must have allow in both the places.
- It basically works as a guardrail
- *VERY IMPORTANT, SCPs are assigned to organizations. Accounts/Organizations inside the organizations INHERIT the SCPS*
