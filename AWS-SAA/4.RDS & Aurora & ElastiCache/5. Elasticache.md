- RDS is used for Relational Databases
  ElastiCache is used for Redis or Memcached
- Caches are in-memory fast databases
- Helps reduce load off of databases for read intensive workloads
	- Common read queries' results are cached
- Helps make application stateless
	- Since state is stored in cache
	- Ex. Session details with a server
- Requires heavy application code changes.
- See this flowchart carefully![[Pasted image 20260413064717.png]]
- Trip to DB is saved, therefore lower load on the DB
- We need to figure out a good cache invalidation strategy so only up-to-date data is stored in cache.
- Sessions Usecase:
- In case we have load balanced EC2 instances, our request might go to different server each time. (in case we have turned off stickiness)
  So we just store the session data(user login data) in a common ElastiCache accessed by all EC2 instances. This way the session data is offloaded to a common storage. So whenever user connects to another instance, that instance can get the user session data from ElastiCache and user will still be logged in.
- It has a lot of similar options like RDS while setting up. (ex. Read replicas, reader endpoints etc.)
  
  
  ### Security
- Elasticache supports IAM Authentication for redis. (Only AWS API level security)
	- This means we cannot connect to Redis using IAM
	- However we can change the permissions of Redis calls (ex. create clusters), using IAM
- Memcached doesnt support IAM
- Redis AUTH
	- It is a password based authentication
	- EC2 uses password -> access Redis
- Encryption
	- In transit (SSL/TLS)
	- Security Groups for network level security
- App → Security Group → Redis AUTH → (optional IAM) → Redis
- Memcached uses SASL authentication, no IAM, no advanced auth

### Caching Strategies
- *Lazy Loading*
	- App → Cache
	  ↓
	  Cache miss → DB → Cache → App
	- Data is only loaded when needed
	- Issue of stale cache (since it is only updated when cache miss happens)
- *Write Through* 
	- Write → DB + Cache (same time)
	- It means cache is always updated
	- No stale data
	- More writes
- *Session Store*
	- Used for login sessions
	- Expires based on TTL

### Use case
- Real-time leaderboard
	- Redis has sorted sets
		- They keep items unique
		- They maintain order (ranking)
	- Example
	  Player A → 100 points
	  Player B → 200 points
	  Player C → 150 points
	  Redis automatically sorts: 
	  1st: Player B (200)
	  2nd: Player C (150)
	  3rd: Player A (100)
	- Redis has very fast access

"Need simple cache, no persistence" -> Memcached
