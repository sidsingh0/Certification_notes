- Automated backups
	- Daily full backup of database happens
	- Transaction logs are backed up every 5 minutes
		- Therefore point in time restore happens to 5 mins ago
	- 1-35 Days of retention, set to 0 to disable automated backups
- Manual backups
	- Triggered by the user
	- Retention of backup for as long as we want
- Trick: in a stopped RDS database, we still pay for the storage. If we want to stop for a long time, we should snapshot and restore later instead
- **Restoring** (Ex. restoring MySQL RDS db)
	- We restore using backups or snapshots
	- We take a snapshot from the desired db 
	- Place the snapshot into s3
	- Restore the backup file into new instance running MySQL

# Aurora backups
- Automated backups
	- 1-35 days (cannot be disabled like RDS)
	- point in time recovery in any time frame
- Manual backups: Same as rds
	- Triggered by the user
	- Retention of backup for as long as we want
- **Restoring**
	- Create a backup of current on-premise database using *Percona XtraBackup*
	- Store the backup file to S3
	- Restore backup file to aurora cluster

### Aurora Database Cloning
- In case we want to test the production Database in a Staging Environment, we use this.
- Create new Aurora DB Cluster from existing one
- It is faster than doing snapshot restore
- Since it uses copy-on-write protocol
	- Initially, the new DB cluster uses the same volume as original DB cluster
	- When updates are made to the new DB cluster data, then additional storage is  allocated and data is copied to be separated
- Very fast and cost effective
