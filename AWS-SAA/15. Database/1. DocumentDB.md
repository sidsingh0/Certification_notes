- DocumentDB is Aurora-like AWS implementation for MongoDB
- NoSQL database
- Store, index, query a JSON
- Same deployment concepts as Aurora
	- Storage grows in increments of 10 GB
	- Millions of requests/second

# Neptune
- Fully managed graph-based database
- Can query upto a **billion** rows, in millisecond latency
- Optimized for dense relationships between data points
	- ex. friends of friends of friends can easily be queried
	- Unlike joins in rdbms which are slow
- Use case: Social Media, Fraud detection system, Wikipedia(knowledge graph), recommendation systems(Customers who bought this als bought)

#### Streams
- Just like dynamoDB streams, Neptune streams is a *real time strictly ordered stream of changes* to our graph data
- It is available via an HTTPS API
- Use case:
	- Send notification when some change happens
	- Backup the streams to S3
	- Replicate data across regions

# Amazon Keyspaces (For apache Cassandra)
- Apache Cassandra, in itself is an open source distributed NoSQL database. 
- Amazon Keyspaces is a serverless managed solution for Apache Cassandra.
- We use CQL(Cassandra Query Language) to query this database.
- It can scale up and down based on our application traffic
- 2 modes:
	- Provisioned with auto-scale
	- On-demand 
- Replicated in upto 3 AZ
- Single ms latency, 1000s of requests/second
- PITR upto 35 days
- Use cases: IOT Device info, time series data
- **Keyspaces** → Excel sheet on steroids (rows + columns, but huge scale)
- **DocumentDB** → JSON storage (flexible, nested objects)
- *Keypoint: > “millions of writes/sec + predictable query pattern”*

# Amazon Timestream
- *Keyspaces was a database that CAN store time series data,*
- *Amazon Timestream, however, is DESIGNED for time series data*
- Fully managed, serverless, scalable and fast DB specifically for time series data
- store and analyze TRILLIONS of events per day
- 1000s times faster and 1/10 cheaper than RDS
- Time stream automatically does:
	- Storage tiering
		- Recent data kept in memory
		- Historical data automatically kept in cost-optimized storage
	- Stores data by time
- Time series analysis function is built in
	- "Give me average of temperature in past 1 hour" -> it will calculate based on persecond data from the sensor
- Use case: IOT devices, realtime analytics

# Redshift
- Data warehouse (OLAP) for TB-PB data
- Used for analytics on huge data warehouses
- DynamoDB and Aurora are not optimized for aggregation queries
- Redshift is DESIGNED for aggregation queries
- Dont use in realtime application
	- high latency for small queries
	- not optimized for frequent writes
	- poor row level operations
	- In above cases us Aurora or DynamoDB
