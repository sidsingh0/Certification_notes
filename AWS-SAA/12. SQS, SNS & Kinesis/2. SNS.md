- SNS brodcasts events to topics.
- Up to 12,500,000 subscriptions per topic 
- 100,000 topics limit
- 2 options to publish
	- Direct Publish for mobile apps (publish to platform endpoint App Store etc)
	- Publish to topic

Flow:
1. Producer produces an event
2. The event is sent to SNS with topic (Using SDKs)
3. SNS publishes the event to the topic
4. All the subscriptions, subscribed to the topic recieve the message
5. There is subscription level filtering available
	1. Filtering is written in JSON
	2. If no filter is there, all the message will be recieved by the subscriber. (BAD)

# FIFO Topics
- Similar to SQS FIFO 
- Order of the messages is preserved within a groupId.
- Deduplication (based on hash of the conent of messages or messageId)
	- If a message with same contentId or messageId is sent after 5 minutes, it is accepted again.

# SNS Security
Encryption:
1. At rest: messages encrypted using KMS Keys
2. In-flight: HTTPS encryption (API Calls made by SDKs use HTTPS)
3. Additional encryption/decryption of messages at client end if required can be set up
Access:
- IAM policies are used to configure Access to SNS
SNS Access policies (Similar to SQS, S3 Bucket policies):
- Allow cross account access to SNS topics
- Useful for allowing other services to publish to SNS topics

# Working with SNS

#### SNS Fan out with SQS
- Amazon SNS is **push-based and does not persist messages long-term**, so delivery can fail if endpoints are unavailable or misconfigured.
- Messages can be lost.
- SQS can send messages only to one subscriber. SNS can broadcast to many.
- Here is the synergy:
	- Producer publishes message to SNS topic
	- SNS fans out message to multiple **subscriptions**
	- Each subscription can be an SQS queue
	- Each queue stores messages durably
	- Consumers read from their respective queues independently
	- ![[Pasted image 20260418051452.png]]
- Advantages:
	- Reliable scaling
	- No tight coupling (since subscribers can be incresaed)
	- Each combination of event type and prefix in S3, only one rule can exist. So if we want to send messages to multiple subscribers based on this rule, we should use fan out.

#### SNS FIFO Fanout with SQS FIFO
- Should be done when ordering, deduplication and fanout is required.
