There are 2 types of queues: 
#### Standard Queue (kw: High Throughput)
- *Duplication can happen*
- *Order is best effort. Not FIFO.*
- Unlimited messages/Unlimited througput, Low latency
- Default message retention: 4 days
- Maximum message retention: 14 days
- Per message can only be 256kb
- Use case:
	- Systems that can handle duplicate messges
	- Logging systems
	- Background jobs

#### FIFO Queue (kw: Order retention, Deduplication)
- First in first out
- *The order of the message is preserved* 
  (Although, Ordering is guaranteed **only within a group, using GroupId**)
	- Example: if we have events in order:
		1.  Order 123 - Created
		2.  Order 123 - Shipped
		3. Order 456 - Paid
		4. Order 123 - Delivered
	   Here, we will use orderId as groupId,
	   Since, only processing order of events inside the same orderId matter
	   If processing of different orderId happens parallely, we do not care.
	   If we use same groupId for different order, everything will be slow due to FIFO.
	   Each message will be processed at time
	   Instead we can parallely process messages of different groups, whose ordering doesnt affect each other.
- *Deduplication mechanism*
- Lower Throughput
- 300 message/s, and 3000 message/second using batch
- Use cases:
	- Transaction Processing systems
	- Reservation Systems
	- Order processing systems

# Features
1. Visibility Timeout (kw: **Retry**)
	1. When a message is picked up by client, it is made invisible for some time, so other clients can't reprocess the message
	2. After processing, we need to delete the message from the queue
	3. Else, visibility timeout will run out, and message will re-appear in the queue
	4. Visibility timeout can not be too short b/c there is a chance of duplicate processing
	5. It cannot be too long, because if processing fails, it needs to re appear. Meaning message processing time will increase by a lot in case of failures.
	6. *Default: 30 seconds*
	7. *In case the process is running out of time, and it wants some more time to complete processing, it can use ChangeMessageVisibility to get more time.*

2. Long Polling (kw: Lower Cost, Higher effeciency)
	1. Sometimes, queues can be empty for a long time
	2. During these times, it is unnecessary to poll queue every 10 seconds 
	3. Flow:
		1. Make API call
		2. No messages in queue
		3. Wait 20 seconds (preferable)
		4. If message appears within 20 seconds, process it
	4. It is better than short polling
	5. **WaitTimeSeconds** can be used to enable this

3. Batch Processing (kw: improve performance, reduce cost)
	1. We can process multiple messages at once
	2. This way number of API calls to get 10 messages(for example) is only 1 instead of usual 10
	3. Maximum messages in a batch recieve can be 10

4. Delay Queue (kw: Process Message after delay)
5. DLQ (kw: Messages that fail multiple times are put here for logging and investigation purposes)

# SQS Security
- Encryption
	- In-flight encryption using HTTPS APIS (accessed through SDKs)
	- At-rest encryption using KMS keys
	- Client can also encrypt and decrypt the messages
- Access control: 
	- IAM roles can be assigned to instance that can access SQS
- SQS Access Policies (Similar to S3 bucket policies)
	- Allow cross-account usage
	- Allow other services to write to SQS (SNS, S3)

# Working with SQS
We use SDK to work with queues. ( sendMessage -> Poll -> process -> deleteMessage)

### SQS with multiple consumers
- In case of multiple consumers, each ec2 instance polls and processes messages concurrently.
- Therefore parallel processing -> high throughput
- Scale horizontally easily
- Best order processing:
	- It may happen that instance 2 processes message faster than instance 1
	- Therefore order of completion of processing is not guaranteed
- Duplicate processing
	- It may happen that worker crashes halfway through
	- Another worker may process it and repeat the steps
	- Therefore we need to code an idempotent system

### SQS with ASG
- We can scale ASG based on the length of the queue
- We create a cloudwatch metric of the queue length 
- Then create a cloudwatch alarm if threshold is passed
- Scale based on cloudwatch alarm

### SQS as a buffer to db writes
- If the load is too big for dbs writes, some data can be lost
- We can introduce SQS as a buffer here.

### SQS to decouple tiers
- Frontend and backend can be decoupled using SQS
