- Serverless database
- Non-relational - NOSQL with transaction support
- Unlimited rows
- Fast and consistent performance
- Never need maintennance and patching (Always available)
- Standard and Infrequent Access 
- * “microsecond latency + massive scale”*
- " eventual consistency + high throughput"

- Just like mongo, each row of the table can look different.
	User1:
	- id: 1
	- name: "John"
	User2:
	- id: 2
	- email: "abc@gmail.com"
	- age: 25
- Every table must have a primary key/composite primary key
- Each row should be less than 400kb
- Datatypes:
	1. Scalar: Number, String, Binary, Boolean, Null
	2. DocumentType: List, Map
	3. SetTypes: String Set, Number Set, Binary Set
1. *You can add/remove attributes anytime without breaking the table*
	- We do not need to update entire table just to change/add columns

### Read/Write Capacity Modes
1. Provisioned (default)
	1. We pay for a fixed number of read and writes per second. [Read Capacity Units and Write Capacity Units]
	2. Used when demand is predictable.
	3. Auto-scaling can be setup later on
2. On Demand (Auto scale up/down)
	1. No need to plan capacity in advance
	2. We pay what we use -> More expensive
	3. Used when demand is unpredictable

# DynamoDB Accelerator (DAX)
- Caching layer on top of DB to reduce traffic
- Default TTL is 5 minutes
- It is fully managed and highly available
- No need to modify existing application code. Compatible with existing DynamoDB apis
- Elasticache caches the aggregate results.

| DAX                   | Elasticache                        |
| --------------------- | ---------------------------------- |
| Only for DynamoDB     | Can be used for any DB or anything |
| Plug and play         | Custom caching logic               |
| Simple read effecient | Complex custom logic               |

- We can delete rows after some time expires using TTL

### DynamoDB Backups

	1. 35 Days 
	2. Point in time restore
2. On-demand
	1. Useful for long-term retention

### DynamoDB with S3
Export
	- PITR for DynamoDB must be enabled
	- **Very Effecient**: *Creating backups and storing them into the S3, does not consume any RCU* (Read capacity units)
	- Use cases:
		- Backups:
			- Export formant: DynamoDB JSON, ION (Advanced)
			- Compliance/Auditing/Logs
		- Analysis:
			- We can export our data from DynamoDB to S3.
			- DynamoDB is not a analytics tool
			- From S3, we can use tools like Athena to perform Analytics
		- ETL:
			- Export: We can export data to S3
			- Transform: Transform the data/make changes to it
			- Load: Load the data back into the DynamoDB. *This creates a new table.*
	- Q: Export DynamoDB without impacting performance
	  A: Export to S3(Using PITR), NOT `scan`
Import
	- We can import data of csv/ION/DynamoDB JSON format from S3 into DynamoDB as a table
	- A new table will be created instead of old one being used
	- **Very Effecient:** *Does not consume WCU (Write capacity units)*
	- Q: Load millions of records into DynamoDB efficiently
	  Answer: Load from S3 and NOT `BatchWriteItem`

### DynamoDB Streams
- **Amazon DynamoDB Streams** captures **changes to items in a DynamoDB table**.
- Retention: **24 hours only**
- Ordered per partition key
- At-least-once delivery
- Serverless (no infra)
- “Trigger Lambda when item changes”  
👉 **DynamoDB Streams**
- "Send email when user registers"
👉 **DynamoDB Streams**
