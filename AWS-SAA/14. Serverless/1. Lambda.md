
What is serverless?
- Developers dont have to manage servers. They just deploy code.

- Costs for lambda are based on:
	- Number of requests
	- Execution TIme
	- Memory Allocation
- In EC2, we pay for idle system aswell, not in case of Lambda
- For long running jobs, constant high load and stateful apps, EC2 may be cheaper
- *By default, Lamda is created outside our VPC*
	- To access resource inside VPC, we must provide VPC id, subnets and security groups to Lambda
	- Then an ENI is created inside subnet, which lamda can use to access VPC

### Lambda Limits
During execution:
- Maximum 15 minutes run time per call
- Allocated ram: 128-10GB of RAM
- Environment Variable size: 4KB
- Concurrent executions: 1000(can be increased)
During deployment:
- compressed deplyment file size: 50MB
- Uncompressed deployment (code + dependencies): 250MB
- env: 4kb

### Lambda SnapStart
- Here is the flow of lambda when it is invoked:
	1. Function initialization
	2. Function Invoked
	3. Shutdown
- Lambda SnapStart allows us to skip initialization part. 
- When we publish a new function, lambda takes snapshot of the memory and disk
- This data is cached with low latency access, therefore used easily next time.
- *It is only available for JAVA 11 and above*

### Customization at Edge
- It is a way to run computations at the edge. (close to the client)
- This allows to reduce the load on the server.
- Fully serverless. Deployed globally (like Cloudfront CDN).

Types:
1. *Cloudfront Function (kw: simple computation, lite, cheap, fast)*
	1. Lightweight functions written in javascript
	2. Useful for less complex stuff
	3. API calls are not done here
	4. Extremely fast (millions of request/second)
	5. Native to cloudfront (Code managed inside cloudfront)
	6. No access to request body
	7. ![[Pasted image 20260419020340.png]]
	8. Use cases:
		1. Modifying CDN content before delivering to the user
		2. Redirect (/home -> index.html)
			1. Generally, server would send 3xx
			2. Client would recieve this response and hit server again to correct path
			3. If server is far, it would take long time for 2 requests, therefore we do it directly at the edge, and only 1 request is required to the main server.
		3. Add/Remove headers
		4. Simple Authentication/Auth
			1. Check for API key in the header
			2. If not available block request
		5. Normalize cache key
		6. Bot filtering
2. Lambda@Edge (kw: complex, authentication, long running, external api calls, slow, large data, expensive)
	1. Different from Normal Lambda
	2. Slower than Cloudfront functions
	3. More expensive
	4. scales 1000s of requests
	5. *Create function in us-east-1 (N. Virginia), automatically copied to other regions*
	6. *Allows Nodejs and Python*
	7. ![[Pasted image 20260419021024.png]]
	8. Usecase:
		1. Third party library usage
		2. File system/API/Network access
		3. Response manipulation, since we have access to the request body
		4. authentication of JWT tokens

# Working with Lambda

### Lambda with RDS Proxy [IMPORTANT]
- If for each request we trigger a Lambda function instance, there will be same number of connections to RDS.
- THIS IS BAD, RDS Cannot handle such huge amounts of open connections
- Therefore we introduce a proxy in between lambda and RDS. 
	- The proxy contains a pool of connections which can be reused. 
	- Therefore number of open connections are reduced to RDS
- *Lambda must be deployed inside our VPC, since proxy cant be made public*

### Lambda Invocation from inside of RDS/Aurora
- For data related events, such as adding row to a table, deletion of rows etc
- We must allow outbound traffic to Lambda from RDS (Public, NAT GW, VPC endpoints)
- DB Instance must have required policy to invoke the Lambda (IAM roles, lambda level policy)

### RDS Event notification
- THese notifications are not about data, but about the RDS instance itself
	- DB Created
	- DB Stopped
	- Snapshot created
	- Maintennance event
	- Failover happened
- THese events can be captured by event bridge and distributed to SNS or Lambda etc.
