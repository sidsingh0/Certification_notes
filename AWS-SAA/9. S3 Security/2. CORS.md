- It refers to cross origin resource sharing
- CORS issues generally means if the domains of two services are different they cannot interact.
- For example, 
	  S3 static hosting enabled website: `http://my-bucket-html.s3-website...`
	  S3 static files in another bucket:    `http://my-bucket-assets.s3-website...`
	  So when website requests files from bucket,
		  GET /images/coffee.jpg 
		  Host: http://my-bucket-assets.s3-website...  
		  Origin: http://my-bucket-html.s3-website...
- To tackle this, we add the following header while requesting for static files:
	- Access-Control-Allow-Origin: http://my-bucket-html...

Q: “Website hosted in S3 cannot load images from another S3 bucket”
👉 99% answer = **Configure CORS on the assets bucket**

# MFA Delete
- If we enable this option, MFA will be required to
	- Delete versions of the object and
	- Suspend Versioning on bucket
- MFA wont be required
	- To enable versioning
	- List deleted versions
- To use MFA Delete, Versioning must be enabled on the bucket 
- Only the bucket owner (root account) can enable/disable MFA Delete

# S3 Access Logs
- We can store access logs for our S3 buckets into a specialized monitoring S3 bucket. 
- Then we can use Data analysis tools
- The target logging bucket must be in the same AWS region
- *DONOT monitor the specialized monitoring S3 bucket. It will create a logging loop, and your bucket will grow exponentially!*

# Amazon S3 – Pre-Signed URLs
*A **pre-signed URL** is just a **temporary, signed link** that lets someone access an S3 object **without needing AWS credentials**.*

- You (or your backend) sign the URL using your AWS credentials  
-  Someone else uses that URL like a normal HTTP. Users given a pre-signed URL inherit the permissions of the user that generated the URL
- We can set expiration for the link
	- Using S3 Console -> 1 minute to 12 hours
	- AWS CLI -> 1 hour to 7 days
	  Q: “Need URL valid for 3 days” -> **AWS CLI**
- Use case:
	- Give access to premium content
	- Dynamic user access -> Let user download files without creating their IAM User account
	- Temporary upload access -> If server knows user has to upload a big file, it gives a presigned url to the user for s3 bucket. This way user can upload files to the bucket directly without going through the server. This saves compute cost and time.
