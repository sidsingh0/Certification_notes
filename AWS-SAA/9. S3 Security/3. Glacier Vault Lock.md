- Adopt WORM(Write one read many) model
- In many industries (finance, healthcare, legal), you **must not delete or modify data for a fixed time** — even accidentally.
- Normal permissions (IAM, bucket policies) are **too flexible**:
	- Admins can change them
	- Data can be deleted
  Vault Lock removes that flexibility.
- Creating a locking policy:
	- You create the Vault Lock policy
	- You test it (there is a testing mode)
	- Then you **LOCK it**
- After locking:
	- ❌ Cannot edit the policy
	- ❌ Cannot delete the policy
	- ❌ Cannot bypass it (even root user)
	This is intentional for **compliance guarantees**
	Protection against tampering

# Object Lock
- **S3 Object Lock**, which is similar in spirit to Glacier Vault Lock—but it works at the **object level inside S3**, not at the vault level.
- *It locks a version of the object, not the entire object. Therefore versioning must be enabled for this to work.*
- It is a version-level WORM protection with flexible (governance) or strict (compliance) retention + optional legal hold.
- Concepts:
	- Compliance Mode
		- Object once locked for a retention period, cannot be deleted or modified.
		- Even CEO can't touch it.
		- Cannot shorten the retention period.
		- Ex. Logs for Auditing (Should not be tampered with)
	- Governance Mode
		- In this case, the lock can be bypassed by privileged users.
		-  `s3:BypassGovernanceRetention`
	- Legal Hold
		- We lock the object indefinitely. No retention period.
		- For example, "Lock CCTV footage till case is closed"
		- Can be turned ON/OFF with permission:  `s3:PutObjectLegalHold`

# Access Point - VPC Origin
A **VPC Access Point** is just an S3 Access Point with one strict rule:
🚫 **You can only access S3 from inside a specific VPC**
No public internet access at all

🏗️ How it works (important for exams)
When you create a VPC Access Point:
You attach it to a specific VPC
AWS creates a VPC endpoint (PrivateLink)
You access S3 using the access point endpoint
 So the flow becomes:
EC2 (inside VPC) → VPC Endpoint → Access Point → S3 Bucket

# S3 Object Lambda
- Use AWS Lambda Functions to change the object before it is retrieved by the caller application
- Only one S3 bucket is needed, on top of which we create S3 Access Point and S3 Object Lambda Access Points
- Use case: 
	- Redacting personally identifiable information for analytics or non- production environments.  
	- Converting across data formats, such as converting XML to JSON. 
	- Resizing and watermarking images on the fly using caller-specific details, such as the user who requested the object.
