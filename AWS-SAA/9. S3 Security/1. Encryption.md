When we talk about encryption, it happens generally at 2 times: At rest and In transport

## Encryption at Rest
To encrypt our objects when placed inside S3.
#### Server Side Encryption
Encryption happens ON the server

1. SSE-S3 (Server Side encryption with S3 Managed Keys)
	- This is the default encryption method when we upload file to a new s3 bucket.
	- The keys used to encrypt the object, are managed, handled and owned by AWS.
	- Encryption type used is **AES256**
	- We send the request with header `x-amz-server-side-encryption": "AES256"` 
	- Flow:
		  Upload → S3 encrypts
		  Download → S3 decrypts automatically
	- ![[Pasted image 20260416061155.png]]


2. SSE-KMS (Server Side encryption with KMS Manged Keys)
	- KMS is Amazon's Key Management Service
	- Here the key is still owned by AWS, however we can manage the keys. (ex. rotation, deletion, disable etc.)
	- *Audit logs via CloudTrail* (How, when, why, who used the key)
	- Flow:
		 Upload → KMS generates key → S3 encrypts
		 Download → KMS decrypts key → S3 decrypts file
	- Required Permissions for downloading:
		- s3:getObject
		- kms:decrypt
	- Even if user has S3 access: Without `kms:Decrypt` → **ACCESS DENIED**
	- Issue with using SSE-KMS:
		- We are stopped by KMS capabilities.
		- Everytime we do generateKeys and decrypt function call (According to the flow),
		  it counts towards our per second quota of KMS calls (can be around 5000-30000)
		- In case this quota is exceeded, the function will fail.
		- Therefore we have to request for more service quota
		- ![[Pasted image 20260416062330.png]]


3. SSE-C (Server Side encryption with Client Managed Key)
	 - Key is entirely managed and owned by the client. 
	 - S3 does not store the keys. Therefore if we lose the key, the data is lost forever.
	 - *HTTPS is Mandatory for this to work*
	 - Client sends the object + key(in the header), this key is then used by server to encrypt the object.
	 - Flow:
		   Upload → you send key → S3 encrypts
		   Download → you send same key → S3 decrypts
	- ![[Pasted image 20260416062744.png]]

#### Client Side Encryption
- In this case there are no subtypes.
- Client sends the encrypted file to be stored, and decrypts it when needed to download.
- Use client libraries such as Amazon S3 Client-Side Encryption Library
- ![[Pasted image 20260416062910.png]]
#### Notes:
- By default, bucket automatically applies SSE-S3 encryption on the objects.
- However we can force other type of encryptions too.
- For this we use bucket policies. 
	- For s3:putObject, refuse any API without encryption headers (SSE-KMS or SSE-C)
	- Bucket Policies are evaluated before Default Encryption
	- ![[Pasted image 20260416063545.png]]
## Encryption in Transit
- We use SSL/TLS for encryption in transit.
- Amazon S3 exposes two endpoints: 
	- HTTP Endpoint – non encrypted 
	- HTTPS Endpoint – encryption in flight (Default)
- SSE-C requires HTTPS only.
- *To force encryption in transit, we can update the bucket policy*: `aws:SecureTransport`, in the condition section. 
	- If a user tries to connect with a bucket with aws:SecureTransport, he will fail
