User Based
	- Attaches to the IAM user/roles
	- "What can this user do?"
	- Ex. User can read/upload to S3

Resource Based
	- Bucket Policies
		- Attached to S3 bucket itself
		- "Who is allowed to access this bucket?"
		- E.x. Anyone can read this bucket, Another AWS account can read
		- Allows Cross account access
	- ACLs
		*- Object Access Control List - Per file access* 
		*- Bucket Access Control List - Per bucket access*
		- ACLs are **legacy + disabled in most modern setups**
		- AWS says DO NOT use ACL, unless mandatory.
- Access is allowed ONLY IF:
	- IAM policy allows OR
	- Bucket policy allows
	- AND ❌ there is NO explicit DENY
- DENY Always takes precedence over ALLOW

- Best practice is to use combination of Bucket Policies + IAM roles
### S3 Bucket Policies
- JSON based policy
- 
  {
	  "Version" : "2012-10-17",
	  "Statement": {
		  "Sid": "PublicRecord"
		  "Effect": "Allow",
		  "Principal": " *  ",
		  "Action": [
			  "s3:GetObject"
		  ],
		  "Resource": [
			"arn:aws:s3:::example-bucket/*"
		  ]
	  }
  }
- Parts of the JSON
	- **Principal:** * => Anyone  => (Public Internet)
	- **Action:** GetObject => Can download files
	- **Resource:** All the files inside the exampleBucket directory
- Use Case:
	- Grant public access to the bucket
		- First change Block Public Access = ON feature to OFF (bucket level/account level)
		- Then change bucket policy to make `prinicipal: *`
		- Make sure that both of the settings are enabled. If even one is disabled, public access will be blocked.
	- Grant cross-account access
	- Force files to be encrypted while uploading.
- Questions:
	- Q1: To allow different account access (Cross account access)
	  A: Bucket policy
	- Q2: To allow access from internet
	  A: Bucket Policy
	- Q3: To allow an IAM user
	  A: IAM Policy
	- Q4: To allow EC2 to access S3
	  A: IAM Roles
	- Q5: To make sure all of the buckets in an account are never accesible to internet
	  A: Enable at account level
