User Based
	- Attaches to the IAM user/roles
	- "What can this user do?"
	- Ex. User can read/upload to S3

Resource Based
	- Bucket Policies
		- Attached to S3 bucket itself
		- "Who is allowed to access this bucket?"
		- E.x. Anyone can read this bucket, Another AWS account can read
		- Allows Cross account access
	- ACLs
		*- Object Access Control List - Per file access* 
		*- Bucket Access Control List - Per bucket access*
		- AWS says DO NOT use ACL, unless mandatory.
- Access is allowed ONLY IF:
	- IAM policy allows OR
	- Bucket policy allows
	- AND ❌ there is NO explicit DENY
- DENY Always takes precedence over ALLOW

### S3 Bucket Policies
- JSON based policy
- 
  {
	  "Version" : "2012-10-17",
	  "Statement": {
		  "Sid": "PublicRecord"
		  "Effect": "Allow",
		  "Principal": " *  ",
		  "Action": [
			  "s3:GetObject"
		  ],
		  "Resource": [
			"arn:aws:s3:::example-bucket/*"
		  ]
	  }
  }
- Parts of the JSON
	- **Principal:** * => Anyone  => (Public Internet)
	- **Action:** GetObject => Can download files
	- **Resource:** All the files inside the exampleBucket directory
- Use Case:
	- Grant public access to the bucket
	- Grant cross-account access
	- Force files to be encrypted while uploading.
- Questions:
	- To allow different account access -> Bucket policy
	- To allow access from internet -> Bucket Policy
	- To allow an IAM user -> IAM Policy
	- To allow EC2 tp access S3 -> IAM Roles
	- To make sure all of the buckets are never accesible to internet -> Enable at account level
