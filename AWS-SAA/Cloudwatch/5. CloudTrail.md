- Used for governance and audit for our AWS account
- Enabled by default
- It gets the history of events related to our account
- We can put cloudtrail logs to cloudwatch logs or s3
- It can be applied to all regions/single region
- *If anything breaks/deletes check CloudTrail logs*
- It can be used to check WHO made changes on oir

Types of Events:-
1. Management Events
	1. Operations performed by our AWS on resources
		1. Changing IAM permissions
		2. Configuring rules for routing data
		3. Setting up logging
	2. *By default cloudtrail logs management events*
	3. We can separate read(That doesnt modify the resource) and write events
2. Data Events
	1. *Not logged by default due to high volume*
	2. Lambda Function level activity
	3. S3 object level activity
3. CloudTrail Insight Events
	1. Cloudtrail looks analyzes our normal management events to establish a baseline
	2. Now based on that baseline, cloudTrail creates events for anamolies
		1. Bursts of AWS IAM actions
		2. Forgetting periodic maintennance
		3. Hitting service limits
	3. Cloudtrail can write these insight events to S3 or Eventbridge events etc.
	4. By default stored for 90 days, can be stored to S3 if more time is required.

- We can create notification if events are triggerred
	- ex. Resource policy for S3 changed send event to event bridge which triggers SNS
