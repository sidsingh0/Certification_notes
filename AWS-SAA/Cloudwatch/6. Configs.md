- Continuously records and evaluates the **configuration of AWS resources**
- Based on the changes and rules, it tells if the service is compliant or not.
- AWS Config is a per region service. Although it can be aggregated over regions and accounts.
- We can store these configs to S3 and then later query them using Athena
- It only checks doesnt block anything
- AWS Managed Config Rules
	- AWS gives you prebuilt rules (75+ and growing):
	- Examples:
		- “S3 buckets should not be public”
		- “EBS volumes should be encrypted”
		- “EC2 instances should use approved types”
	- 👉 You just enable them, No coding needed.
- We can also make our own rules
- These rules can be evaluated when
	- Rules change
	- Over a period of time

Keywords:
- “compliance checking”
- “configuration history”
- “resource changes over time”
- “audit what changed”
- “detect misconfigurations”

Questions that can be solved by AWS Config: 
- Is there unrestricted SSH access to my security groups? 
- You want to know if a security group ever allowed public SSH access in the past.
- Do my buckets have any public access? 
- You need to ensure S3 buckets are never public.
- How has my ALB configuration changed over time?

# Remediation
- Config finds problems but doesn't fix them.
- Remediation means automatic fixing of these issues.
- Flow:
	- Config detects non-compliance
	- Fix is triggerred using AWS Systems Manager Automation Documents
	- Fix is applied
	- If resource is still non compliant
	- We can retry remediation script with max retries

#### AWS Systems Manager Automation Documents
- It is a predefined runbook(script) that tells AWS what to do.
- Example:
	- Disable IAM key
	- Stop EC2 instance
	- Fix security group rule
- AWS provides some AWS-managed documents (ready to use)
- You can also create your own custom ones
