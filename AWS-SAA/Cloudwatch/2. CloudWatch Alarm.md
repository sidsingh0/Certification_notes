- Alarms are used to notify when a metric reaches a threshold
- Possible values:
	- OK
	- ALARM
	- INSUFFECIENT DATA
- Various Sampling methods:
	- %
	- Min
	- Max etc.
- Period:
	- Length of time over which a metric is sampled
	- High resolution: 10s, 30s or multiples of 60s
- Example use case:
	- Stop, terminate, reboot an EC2 Instance
	- Trigger Auto Scaling
	- Send notification to SNS

___

#### Composite Alarms
- Monitor multiple metrics to trigger alarms
- Use OR AND conditions to club metrics
- "Helpful to reduce alarm noise (Too many alarms)"

___

#### Recovery for EC2 instance
- AWS generally checks 2 layers for our EC2 instance
	1. Instance Status Check
		1. OS running or not?
		2. System hanged or stuck?
		3. Application responsive or not?
	2. Underlying Hardware Check
		1. Is the physical machine okay?
		2. Is there hardware/network/power failure?
- Based on these 2 checks, alarm is triggerred which can notify the admin
- Similar usecase: Same Private, Public, Elastic IP, metadata and placement group.

___

#### To Test Alarm
- Set alarm state to "ALARM" and see if it is triggerred.

___

#### Trigger Alarm Based on Cloudwatch Logs
- You can create CloudWatch alarms that watch for specific patterns in your logs, instead of just numeric metrics like CPU or memory.
- For ex: "If ERROR appears in logs for more than 5 times in a minute, trigger the alarm"
