AWS pushes for Hybrid cloud (Cloud + On premise architecture) so that the 
- transition to cloud is easier
- Disaster recovery (backups)
- compliance and 
- security is maintained.

The problem is, S3 is a proprietary technology. How do we expose S3 to our on premise architecture? We use Storage Gateways.

### Storage Gateways
Bridge between on premise and aws cloud infrastructure. These gateways are installed on on-premise servers.

Types: 
1. S3 File gateway
	1. *We connect our on-premise server to S3 File gateway, using SMB/NFS protocols.*
	2. This gateway connects with S3 over HTTPS.
	3. The gateway also caches files for fast response
	4. *Only allowed for Standard, Standard IA, One Zone IA and Intelligent Tiering.*
	5. To use glacier, we can use lifecycle policy.
	6. We give IAM role to the File Gateway with permission to conenct to S3.
	7. SMB has AD integration for auth.	   ![[Pasted image 20260417130949.png]]

2. FSx File Gateway 
	1. Connect natively to FSx for Windwos Server
	2. Local cache for frequently accessed data
	3. Same native windows capabilities (SMB, NTFS, AD)
	4. Useful for group file shares and home directories

3.  Volume Gateway
	1. Let's say our company server requires more storage
	2. We make the server feel like it has a hard disk attached to it with low latency
	3. *But in reality, we use AWS with ISCSI protocol (Internet Small Computer System Interface) to connect to S3.*
	4. Also creates EBS snapshots for recovery
	5. Used when storage is required (ex. apps, datbases on premise)
	6. Types
		1. Stored: Volume stored locally, available offline, syncs with aws after sometimes
		2. Cached: Fast frequently accessed files

4. Tape Gateway
	1. Your system thinks: “I am writing data to tapes for backup”
	2. But actually: Data goes to AWS Glacier
	3. Use case: 
		- backups 
		- long-term storage
		- compliance
	4. Do not use in case of running applications.
	5. Also uses ISCSI

### Transfer Family
- Storage gateways are for internal systems.
- Storage gateways use iSCSI, SMB, NFS, however, older systems require, SFTP, FTP, FTPS.
- External vendor doesnt want to mount NFS share, install my gateways etc.
- So we have Transfer Family, which allows, older protocols like SFTP FTP and FTPS to upload files to S3 and EFS.
- It is a file transfer tool unlike storage gateways which are hybrid storages logically.
- Fully **managed**
- Integrate with existing authentication systems (Microsoft Active Directory, LDAP, Okta, Amazon Cognito, custom) 
- Usage: sharing files, public datasets, CRM, ERP, Suppose bank shares details daily and we need to upload to the S3 bucket

### Datasync
- Used to transfer large amount of data.
- AWS to AWS resource (One storage to another) -**Requires NO agent**
- On premise / other clouds -> AWS - Requires an agent
- Each agent can use upto 10 GBPS of badwidth
- It is available for S3(Standard + Glacier), FSx, EFS 
- File permissions + Metadata is retained. (No need to create policies again)
