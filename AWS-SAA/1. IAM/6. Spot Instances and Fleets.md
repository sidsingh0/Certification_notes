### Spot Instances
- We define a max-spot-price and we get an instance whenever 
  `max-spot-price < current-spot-price`
- If max-spot-price < current-spot-price, we can choose to terminate or stop the instances.
- Spot block (now discontinued) used to allow us to reserve an instance for (1-6 hours) without interruptions, although in very rare cases, we could lose our instance
- current-spot-price is dependent on Availability Zone (ex. asia-northeast-1a)
- Spot request:-
	- Instance type
	- Number of instances
	- max-spot-price
	- *Request type: ONE TIME or PERSISTENT*
		- one time: as soon as our requirements are fulfilled, spot request goes away
		- persistent: as soon as our requirements are fulfilled and we get a instance, and then we lose the request, it wont go away, that request will stay there to find new instance once conditions are fulfilled.
	- valid from, valid until (can be infinite)
	- ![[Pasted image 20260408021939.png]]
	- We can only cancel spot instance requests that are open active or disabled
IMPORTANT NOTE:
To terminate a spot instance forever this is the correct order:-
Cancel the spot request -> Terminate the instance

### Spot fleets
- Set of Spot Instances + (optional) On demand instances
- Spot fleets tries to meet target capacity with price constraints.
- We need to define launch pools(single/multiple) for this.
- A launch pool is basically a set of unique (instanceTypes + OS + AZ)
- Spot fleet will stop launching instances when budget or capacity is reached.
- Strategies:-
	- `LowestPrice`: launch instances with lowest price(short workload)
	- `diversified`: launch instances distributed across all pool (if one pool is sold out, others might remain -> high availability -> long workloads)
	- `capacityOptimized`: pools with highest capacity
	- `priceCapacityOptimized`: (recommended): pools with highest capacity available, and then select pool with lowest cost.
