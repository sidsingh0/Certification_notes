Identity and Access Management
- It is a *global service* used to manage users and their permissions.
- We can create users for our organization here.
- These users can be assigned to groups.
- A user can be assigned to *zero or multiple groups*.
- Groups only contain users, and *not other groups.*
- Essentially, we create groups so that people with same role/project/teams can have common permissions. This way we don't have to manage each user's permissions individually.
- ![[Pasted image 20260403005804.png]]
- We can assign permissions to groups/users with JSON documents, called policies.
- Assigning permission directly to a user(not in group) is called *inline policy*.
- *Always apply the principle of least privilege.
- Do not use root user.
- To figure out if you are using IAM user or Root user, click on the account name on top right side.
	- If you only see account number, it means you are using root user 
	- If you see both account number and IAM username, it means you are using IAM user.

## IAM Roles
- Sometimes, AWS services, might need to access AWS on our behalf.
- In this case, we give permissions to the service using *Roles*
- For example, ec2 wants to perform actions on aws. So we will assign role to ec2.

## IAM Security Tools

### 1. IAM Credentials Report (Account level)
- Generates and lists all the account's users and their credential management status
- It contains things like:
	- user creation date
	- MFA enabled/disabled
	- password changed date
	- last region used
	- last service used etc.
- It can help find out which users are keeping up with best credentials management practices. And which users we should be focusing on from a security standpoint.

### 2. Last Accessed (User Level)
- It shows the service permissions and service accessed by the user.
- It helps to identify what permissions are not used by the user and remove those permissions. (principle of least privilege)
