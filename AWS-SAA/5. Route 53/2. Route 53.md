- It is a domain registrar.
- It is highly available, scalable, fully managed and Authoritative DNS
	- Authoritative means we can update the DNS records as a customer
- Ability to check health of our resources
- The ONLY AWS service which provides 100% availability SLA.
	- SLA = Service Level Agreement. It’s AWS’s guarantee about uptime.
- 53 is the traditional DNS port.

### Records
- In route 53, we define a set of records.
- These records tell how we want to route traffic for a certain domain.
- Each record contains:-
	- Domain/Sub domain name
	- Record Type: A, AAAA, CNAME
	- Value: 12.32.54.78
	- Routing Policy: How route 53 records to the queries
	- TTL: Amount of time our record is cached at the DNS resolvers.

### Hosted Zone:
- Container of DNS Reccords
- Two types:
	- Public Hosted Zone
		- Works on Internet
		- Anyone can query
		- example.com -> public EC2 IP
	- Private Hosted Zone
		- Works only inside the VPC
		- Invisible to the internet
		- Generally used for internal sites & micro services
		- Internal services → private IP
- We pay 0.5$ per hosted zone used per month

### Record Types:
1. A: maps hostname to a IPv4
2. AAAA: maps hostname to a IPv6
3. CNAME: maps hostname to another hostname
	-  The target is a domain name which must have an A or AAAA record
	-  Can't create a CNAME record for the top node of a DNS namespace.
		- The top node = your root domain, like: example.com ✅ (top node), www.example.com ❌ (NOT top node),api.example.com ❌
4. NS: Tells the world → “These are the DNS servers for this domain”. It is created automatically.

### Debug
- To find out the A record/server Ip of a domain, we use `nslookup (domain-name)`
- To see TTL as well as A record, `dig (domain-name)`

### TTL
- When we request a website for ex. google.com
- Our request first goes to DNS server. 
- DNS returns a response with the IP of google.com and TTL = 300s
- We cache this domain:ip mapping and connect to the ip
- So for next 300s if we request the same domain (google.com), our client will not request the DNS server, instead it will directly connect to ip that was cached for that domain
- We do this, since we dont expect DNS records to change a lot.
- If TTL is too high (24h), less requests on Route 53(¥), however, if we update DNS records, it will take 24 hours for all users to see updated website.
- If TTL is too low(60s), more requests on Route 53(¥¥¥), however, users will see the updates to records instantly
- Therefore, when we want to change a record, we generally set TTL to a low value. Once we know that the record change has been propagated to the users, we increase the TTL value back up.
- TTL is manadatory for all the records. EXCEPT ALIAS.


### CNAME vs ALIAS
- When we have a resource like load balancer or cloud front, AWS exposes a hostname instead of IP.
- To use this hostname with our custom domain we purchased, we need to update the CNAME records or ALIAS records.
- *But CNAME only works for non-root domain name (ex. sid.example.com✅, example.com❌)*
- ALIAS: 
	- Route 53 specific. ALIAS record is an extension of DNS capabilities. (meaning not all DNS have it)
	- Points a hostname to an AWS resource.
	- *Works for both Root domain and Non-root domain (ex. sid.example.com✅, example.com✅)*
	- Free of charge
	- Native health check
	- ALIAS records are always of A/AAAA type, and its TTL is managed by AWS Route 53.
	- It automatically recognizes, change in the underlying IP of the resource and updates the record.
	- Target resources for ALIAS record:
		- ELB
		- Cloudfront
		- API gateway
		- S3(bucket being used as a website)
		- Route 53 record added in the same hosted zone
		- Elastic Beanstalk Environments
		- VPC Interface endpoints
	- *We CANNOT add alias record for EC2 DNS Name.*

### Health Checks
- **Route 53 Health Checks = Decide whether a DNS record should be used or ignored**
- So that if user requests for IP of the domain, they aren't returned an IP of a dead server.
- In case we have 2 ALB in 2 different regions, and one region goes down, DNS records can redirect users to ALB in the other region. i.e. DNS failover is achieved.
- In case all the servers are down, then Route 53 still returns a response -> with bad IPs (Since it has to return a response).
	- Although in this case we can return a secondary failover record located in another region for better handling.
- But incase only some servers are down, route 53 only returns healthy ips.
- HTTP Health checks are only for public resources
- Types:
	1. Endpoint Health Check
		- Monitors a Public endpoint. (ALB interfaced to internet, EC2 public IP)
		- 15 Global AWS health checkers send request.
		- They check HTTP/HTTPS/TCP
		- Endpoint must return 2xx/3xx, anything else means unhealthy
		- Endpoint's response body content can also be checked to declare it healthy/unhealthy. (Ex. Look for "OK" in first 5120 bytes of the body)
		- If >18% of the checkers say that endpoint is healthy, overall result is declared as healthy
		- Interval is by default 30s. Fast check is 10s(More expensive).
		- *Our endpoint must allow traffic from the Route 53 Health Checker IP ranges*
	2. Cloudwatch Alarm Health Check
		- This has the most control
		- Route 53 health checkers work outside the AWS VPC -> They can only access public endpoints
		- So private endpoints can't be accessed.
		- Solution:
			- Create **CloudWatch Metric** - Example: CPU, memory, custom metric
			- Create **CloudWatch Alarm** - Trigger when metric is bad
			- Link alarm to Route 53 health check (Health Checker monitors this alarm)
			- Alarm = OK → healthy ✅
			- Alarm = ALARM → unhealthy ❌ → record not returned to user 
	3. Calculated Health Check
		- This is used for a higher level understanding of the system. Tells us how system is performing.
		- Combines results of multiple health checks into 1.
		- It doesn't check server directly, instead it checks other health checks.
		- Our instances with public IPs already have health check, we create a parent health check. (Which evaluates based on AND, OR operators)
		- LIMIT: Maximum 256 health checks
- Health check metrics are stored as CloudWatch Metrics
- To intentionally fail a health check(for testing), we can block HTTP port for that instance.
### Routing Policies
- It defines, how route 53 responds to DNS queries.
- Word "routing" here is not actually routing.
	- Route 53 does not route any traffic, it only responds to DNS queries.
- These policies, can be applied to any record type. But generally used with A/AAAA records.

Supported Policies:
1. Simple: 
	- Typically route traffic to a single resource.
	- We can return multiple values too.
	- Client will choose a random value to connect to the resource.
	- If Alias record is enabled, we can only specify one AWS resource.
	- *We cant use health checks here*.
2. Weighted
	- Suppose we have multiple values (IPs)
	- We can assign each IP a weight, so DNS will distribute the responses using this weight.
	- For example:
		  IP 1 = 70
		  IP 2 = 30
		  Then 30% of the requests to DNS will get IP2 as response and 70% will get IP1 as response.
	  - If one of the record's weight is set to 0, that IP is not returned.
	  - If all of the record's weight is set to 0, the responses are equally distributed.
	  - We add multiple records with different weights, to the same hostname, to make this happen.
	  - Use Case: Load Balancing between regions, testing by sending small amount of traffic to it
  3. Latency Based
	  - Redirects to the resource with lowest latency.
	  - Useful when latency is important.
	  - Latency is dependent on traffic between users and AWS regions
	  - i.e. If we have users in mumbai region but traffic is lower in tokyo region(therefore latency is lower), the users are directed to Tokyo region's aws resource
	  - Can be associated with health checks (has failover capability)
	  - We add multiple records with different regions, to the same hostname, to make this happen.
  4. Failover Based
	  - For a hostname, there can only be 2 records, in this case. Primary and Secondary.
	  - Primary EC2 instance, mandatorily needs to have health check setup.
	  - When this health check goes unhealthy, Route 53, automatically returns Secondary Record. It can also have a health check if we want.
	  - So these are the steps:-
		  - We create a health check for the primary ec2 instance.
		  - Then we create the record for primary ec2 instance using our health check.
		  - With same hostname, we create secondary failover, here health check is optional.
  5. GeoLocation
	  - Based on where user is actually located
	  - Specify location by Continent/Country/US state. In case of overlap, more precise area is selected.
	  - A default location is required. In case all the locations don't match.
	  - Use case: 
		  - Localization: when we put hm.com in japan, it automatically opens japanese website
		  - Content Restriction: when we put tiktok.com in india, it automatically redirects us to page saying Tiktok is banned in India.
	  - While creating the records with routing policy as `geolocation`, always create a default record with location set as `default` as a fallback.
6. GeoProximity
	  - Similar to Geolocation, routing based on where user is actually located
	  - We have a extra parameter called Bias here.
	  - *Ability to shift more traffic towards a resource* 
	  - To expand a geographic location, set bias as -> 1 to 99 -> more traffic to this region
	  - To shrink a geographic location, set bias as -> -99 to -1 -> less traffic to this region
	  - Resources can be:
		  - AWS resources (Specify AWS region)
		  - Non-AWS resources (Provide longitude, latitude)
	  - ![[Pasted image 20260414033638.png]]
	  - We should use Route 53 traffic flow to use this feature.
  7. IP based routing
	  - Routing based on client IP address
	  - We provide a list of CIDR blocks to route users to corresponding instance
	  - Use case: Reduce network cost, optimize performance
	  - We can route a certain ISP to a certain endpoint based on this
	  - Try to include a default fallback
  8. Multi Value Routing
	- Use when routing to multiple resources
	- Route 53 returns multiple values/resources
	- *Associate with health check (This is why it is different from simple routing)*
	- We create multiple records in this case too
	- Unhealathy records are not returned
	- Upto 8 healthy records can be returned for each Multi Value query
	- Not a substitute for having an ELB

### Route 53 Resolver
- It is the DNS server inside our VPC.
- Every EC2 instance, automatically uses this.
- It can resolve 
	- Private hosted zones (e.g. `db.internal`)
	- Public domains (`google.com`)
- Now lets say we have on premise architecture with our own DNS server and internal routes.
	- AWS CANNOT resolve on-prem DNS ❌
	- On-prem CANNOT resolve AWS private DNS ❌
- Therefore we create a hybrid DNS to make both sides talk to each other.
- First we need to set up VPN or Direct Connection between VPC and On-premise
- Now we have 2 endpoints:-
	- Inbound Endpoint (On Premise -> AWS resources)
		1. On-prem server asks:
		    - “What is `app.aws.internal`?”
		2. Goes to:
		    - On-prem DNS server
		3. That DNS server forwards request to:
		    - **Inbound Endpoint in AWS**
		4. Inbound endpoint asks:
		    - Route 53 Resolver (in a hosted zone)
		5. Returns answer back
	- Outbound Endpoint (AWS -> On-Premise)
		1. EC2 asks:
		    - “What is `db.corp.local`?”
		2. Route 53 Resolver sees:
		    - “I don’t know this domain”
		3. Forwards request to:
		    - **Outbound Endpoint**
		4. Outbound sends it to:
		    - On-prem DNS server
		5. Answer comes back
