- Querying tool for data stored inside S3.
- It is managed and serverless
- We are charged $5/Tb scanned
- Uses SQL to query files
- Supported formats: 
	- JSON
	- CSV
	- Parquet
	- Arvo
	- ORC
- *We use this with AWS Quicksight for creating dashboards*
- Use case:
	- Analytics
	- Analyzing VPC logs
	- Cloudtrail trails
	- Business intelligence
	- If question says analyze data in S3 using serverless SQL -> Athena

#### To increase Athena Performance
- Use columnar data instead of row 
- Shard our S3 bucket
	- Example: s3://athena-examples/flight/parquet/year=1991/month=1/day=1/
	  Format:  s3://bucket-name/pathToTable/partition_column=value/partition_column=value...
- Compress data for smaller retrievals
- Use files larger than 128mb > so overhead is not wasted

#### Athena Federated Queries
- We can query multiple data sources (Relational, Non Relational, custom data sources)
  using SQL
- *Data source connectors are run on Lambda to run federated queries*
- Results are stored back into S3

# Redshift
- Based on PostgreSQL
- It is faster than Athena (because of indexes)
- It is used for OLAP (Analytics not OLTP-transactions)
- *It uses Columnar Data instead of row based*
- We can perform queries using SQL
- 10x better performance than other warehousing tools
- Similar integration(as Athena) with BI tools like Quicksight or Tableau 
- Two modes: 
	- Provisioned cluster 
	- Serverless cluster

#### Redshift Cluster
- A **Amazon Redshift cluster** is a **group of machines (nodes)** that work together to store and process your data.
- Architecture:
- ![[Pasted image 20260421051834.png]]
- Leader node(basically manager) is responsible for
	- Getting queries
	- Parsing it
	- Planning it
	- Sending to worker nodes
	- Combining results from compute nodes
- Compute nodes(employees) are responsible for:
	- Storing data
	- Executing queries
	- Returning results to master
- Why this architecture? Because we can increase parallelism, compute power and storage this way by increasing compute nodes.

#### Redshift backups and DR
- Snapshots are stored in S3 bucket but the important thing is, they are incremental and not full snapshot. This way memory is saved.
- They are restored into new cluster and not the old one.
- They can be manual, or automatic (every 1-35 days)
- We can copy snapshot to another region for DR and region failure
- Multi AZ might be allowed

#### Loading data into Redshift
- From Amazon Kinesis Firehose (streaming data)
- From S3 using copy command (using internet or amazon VPC for faster speed)
- From EC2 instance (should be done in batches)

#### Redshift Spectrum
- Using this we can read data directly from S3, without loading it.
- Only requirement: HAVE REDSHIFT CLUSTER to start query. 
- Each worker node is connected to spectrum node which are connected to s3

# Amazon OpenSearch
- Google search for your application data (Since normal DBs are bad at searching data)
- DynamoDB only allows searching based on indexes and primary key, opensearch allows any field
- Opensearch can do
	- Fuzzy matching
	- Partial matching
	- Search by field
- We do not need to replace our DB, we just add opensearch layer at the top,
- So now the flow is:
	- We insert data into the DB
	- Then we insert same data into Opensearch (which indexes it)
	- Finally we can search in openSearch directly
- Does not natively support SQL, it supports JSON/Search terms
- We get dashboard free with it
- Input can be:
	- Application data
	- Kinesis Firehose
	- Cloudwatch Logs
	- IOT sensor data
- Two modes
	- Managed cluster: Fixed cluster size, aws manages availability scaling and patch
	- Serverless cluster: we just send data, scaling + everything else is managed by aws
