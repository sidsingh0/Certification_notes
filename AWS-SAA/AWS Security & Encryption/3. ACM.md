- We can create and deploy TLS certificates using this
- **Auto renewal for certificates**
- Public Certificate and Private Certificate Both are supported
- Public Certificates are free of charge
- Integration with:
	- Elastic Load Balancers (CLB, ALB, NLB) 
	- CloudFront Distributions 
	- APIs on API Gateway
- **CANNOT USE WITH EC2 (Since AWS will have to share secrets then)**

#### Requesting Public Certificate
- To get a certificate for TLS, we need to request for it to AWS
- Request cert → Validate domain → Issue cert → Auto-renew forever
	1. We tell the ACM what should websites should be secured with TLS
		- ex. Wildcard(\*.example.com), domain.com, sid.domain.com etc.
	2. Prove we own the domain
		- We have 2 options for this
			1. Add a **CNAME record** in DNS (e.g., Route 53)
				- Example: CNAME `_yyyyy.acm-validations.aws` 
				- ACM checks it automatically
			2. Click a link in the email (Not preferred, manual)
	3. AWS issues cert in minutes to hours
	4. Automatic renewal
		1. Happens every 60 days automatically if we used DNS method
		2. Else, we have to click verification link again

#### Importing a Public Certificate
- Instead of requesting a Public Certificate,
- We can generate a certifcate outside of ACM and import it
- However, this method doesnt have automatic renewal
- ACM sends daily expiration events starting 45 days prior to expiration • The # of days can be configured • Events are appearing in EventBridge
- *AWS Config has a managed rule named acm-certificate-expiration-check to check for expiring certificates (configurable number of days)* -> To check for compliance

#### Integrating ACM with API Gateway
We remember that API gateway had 3 different kinds of endpoints (Edge optimised, regional, private within VPC)
- If we have a custom domain name in API Gateway, we need to set up ACM
1. Edge optimized endpoints
	- Requests to API Gateway are routed through edge locations to internal AWS Network to the API Gateway (better latency)
	- API Gateway lives in only one region
	- *For TLS to work ACM Has to be deployed in US-EAST-1 region*
	- Then setup CNAME or (better) A-Alias record in Route 53
2. Regional Endpoints
	- This is used for regional users
	- ACM has to be in the same region as the API Gateway
	- Then setup CNAME or (better) A-Alias record in Route 53
