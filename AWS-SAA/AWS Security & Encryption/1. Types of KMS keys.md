1. AWS Owned keys - 
	1. Default keys used in background when we enable encryption feature. 
	2. We cannot see the key. 
	3. Example: S3 bucket keys, SNS, SQS etc.
2. AWS Managed Keys - 
	1. Same as owned key, but we can see the keys in our account
	2. keys are linked to our account
	3. This is useful for transaprency and auditing
	4. We cannot create or generate these. They are created by amazon themselves
	5. Example: aws/service-name like aws/rds, aws/ebs
3. Client Managed Keys
	1. Client uses KMS to generate keys
	2. Client can still NOT see the keys
	3. However client can manage the keys like delete, rotate, disable etc.
4. Client Owned Keys
	1. Client owns and creates the key on their own
	2. They use KMS infrastructure to store and manage the key
	3. THerefore companies can set their own rules for key generation
	4. Useful for compliance/hybrid security models
	5. Problem is we need to rotate keys manually, that is very problematic

Important Note: Never store your keys in plaintext in your txt files or source codes. Encrypted keys are safer, since KMS is available via API call, we can just get encrypted version of the keys to store in plain text.

### Automatic key rotation
- AWS Managed key auto rotates every 1 year
- Client Managed Key rotates every 1 year (But needs to be enabled)
- Client Owned Key has to be rotated manually using Alias
	- “Using alias” means your application always references a stable name, while AWS lets you change which actual key is behind it.

### KMS Key Policy (Resource level access control)
- Similar to S3 bucket key
- However in S3 bucket key if IAM allows a user to access the bucket and there is no bucket policy, access is **allowed**
- Whereas, Key Policy and IAM access both must exist and be SET to ALLOW
- Default key policy: Allow root user the full access to the keys, it is created by default if we dont allow it
- Custom Key Policy: Similar to S3 bucket keys, we can allow who can access/administer the keys, also useful for cross-account access. ↓↓↓

### Copying snapshot across accounts 
- We require custom key policy for this
- We encrypt the snapshot with our Account 1 key
- Then, we change the policy to allow Account 2 to access this key
	- `kms:CreateGrant` : meaning account B can let other resources use this key without changing key policies
	- `kms:Decrypt`: account B can call decrypt API 
- Share the encrypted snapshot with Account 2
- *Account 2 creates a copy of the snapshot*
	- This automatically triggers decrypt 
	- Then recrypt using account B keys
- Now we can create a volume with the snapshot

### KMS Multi Region Keys
- Keys in KMS are region locked, We cant use asia south keys in asia pacific
- We can enable Multi Region Keys so:
	- Encrypt in one Region and decrypt in other Regions
	- No need to re-encrypt or make cross-region API calls
	- Global client side encyption
	- encryption in global DynamoDB Global Aurora
- Components:
	- Multi-region Primary key
	- Multi-region Replica key
	- These keys are in sync with each other
	- KMS multi region keys are not global
	- Each key is independent
		- Can be managed independently (enabled, disabled etc.)

### DynamoDB Global Tables + KMS Multi Region Keys + Client side encryption
- We don't do general Server Side Encryption in case of DynamoDB because, 
	- DynamoDB encrypts data **after it receives it**
	- KMS only protects data inside of AWS
	  But,
	- DynamoDB can see our encryption key before encryption
	- process queries on unencrypted attributes
	- if dynamoDB encrypts, it needs to encrypt entire data and not just one attribute of it
- Now, lets say client wants to encrypt certain attributes of the row being inserted into a Global DynamoDB Table
- He can encrypt the attributes (Like password, PIN, SSN) by storing a client key in his region's KMS
- But the problem is when users in different region access the GlobalDynamoDB and try to decrypt the data, they will have to make a long global call to the region which contains the KMS key
- Therefore we use KMS multi region keys
- So the users where there is KMS Multi Region Key available, they will have very low latency.
- We use **Amazon DynamoDB Encryption Client** on client side for encryption and decryption
- This way even database admins cannot see the encrypted data
- *SAME PROCESS FOLLOWS FOR AURORA GLOBAL TABLES with ***AWS Encryption SDK**


### S3 Replication Consideration
- Replication is copying of files to a bucket in another region for DR purposes
- Unencrypted files -> Copied Normally
- Encrypted with SSE-S3 -> Handled Internally
- Encrypted with SSE-C -> We have to provide key on each request. Very Tricky.
- Encrypted with SSE-KMS
	- Setup is required
	- S3 has to decrypt the object with its KMS key  
	- then target bucket has to re-encrypt the object with its own KMS key
	- S3 replication does NOT include KMS objects by default. We have to configure it in the settings.
	- Tell Target bucket → use THIS KMS key
	- We need to give permissions in 2 places
		- KMS key policies 
			- Allow S3 replication role to decrypt with source key
			- Encrypt with target key
		- IAM Role Permissions
			- kms:Decrypt → source key
			- kms:Encrypt → target key
	- *During this KMS API may throttle, we have to ask for service quota increase*
- Encrypted with Multi-Region SSE-KMS
	- STILL encrypt then rencrypt
	- It doesnt copy as is

### AMI Sharing Process Encrypted via KMS
- AMI is just metadata + EBS snapshot
- EBS Snapshot is encrypted in the source account with its KMS
- We share the AMI with account 2
	- ModifyImageAttribute → add Account 2
	- But now the AMI is visible but not accesible since it is encrypted
- We update KMS Keys Permission
	- Account 2 -> kms:Decrypt, kms:CreateGrant etc
- In Account 2, the IAM role/user should have the following permissions:
	- kms:Decrypt
	- kms:createGrant
	- kms:DescribeKey
	- kms:ReEncrypt* → if copying/re-encrypting
- Now, to launch EC2 with AMI, we have 2 options
	1.  Use source KMS key (not ideal) - since it still depends on account 1's KMS Key
	2. Re-encrypt with own KMS key - happens automatically
	   Decrypt (source key) → Re-encrypt (target key)
