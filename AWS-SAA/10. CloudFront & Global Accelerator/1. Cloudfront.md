- We have more than 200 edge Points of Presence(Edge locations) all around the world.
- Cloudfront utilizes these points of presence to serve as a CDN.
- It reduces READ latency. Since content is caches at these points of presence.
- Cloudfront comes integrated with AWS Shield -> DDoS protection + AWS Web App Firewall

### Origins
1. S3 bucket.
	1. For distributing files and caching them at edge locations
	2. Enhanced Security -> CloudFront Origin Access Control. (OAC) + already existing Bucket policies
	3. Cloudfront can be used as an ingress to upload files to s3
	4. Private network is utilized between S3 and edge locations.![[Pasted image 20260416194332.png]]
2. Custom Origin (HTTP)
	1. Application Load Balancers
	2. EC2 Instances
	3. S3 Static Website
	4. ANY HTTP Backend we want

#### Cloudfront vs S3 Cross Region Replication
Both reduce latency for end users

| Cloudfront                                                                              | Cross Region Replication                                                                                                                         |
| --------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| Global network                                                                          | Region wise setup                                                                                                                                |
| Edge locations can be used as ingress for S3 buckets                                    | Read only                                                                                                                                        |
| Good for static files that need to be available everywhere (ex. main.css, main.js etc.) | - Good for dynamic content. Since commonly changed and updated files are copied instantly to different regions.<br>- Dev, Test environment setup |
| Files are cached for TTL                                                                | Files updated in real time                                                                                                                       |

### Cloudfront - ALB or EC2 as Origin
- EC2
	- If we keep EC2 as origin of for the Cloudfront, it's IP has to be made public. 
	- Allow Public IP of Edge Locations into EC2 security group
- ALB
	- We can use ALB in front of EC2 instances, this way IP of instances are hidden.
	- We allow Public IPs of edge locations into ALB and ALB's security group into EC2

### Geo-restriction
- We can restrict people of certain countries to not be able to access our cloudfront cache.
	- AllowList: Allow your users to access your content only if they're in one of the countries on a list of approved countries.
	- BlockList
- The country is calculated based on third party IP address's location resolver.
- Use case: Copyright laws to access a content

### Pricing
1. Price class All: Most Expensive, Highest performance, All regions.
2. Price class 200: most regions, but excludes the most expensive regions.
3. Price class 100 : Cheapest. Includes the cheapest regions.

### Cache invalidation
- In case we update our backend, our cache might not be updated at cloud front edge locations.
  Due to TTL not being expired.
- We can invalidate the *Cloudfront Invalidation* completely/partially to solve this.
- You can invalidate all files ( * ) or a special path (/images/ * )
