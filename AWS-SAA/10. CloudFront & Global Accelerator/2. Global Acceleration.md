- When we create an application, users want to access it directly.
- However, their request goes through the public internet. Therefore they have a lot of **hops** from server to server.
- We try to minimize this latency by using AWS's private network instead.
- Unicast IP: one server holds one IP address
- Anycast IP: all servers hold the same IP address and the client is routed to the nearest one

### Features
- Global accelerator sits in front of:
	- ALB
	- NLB
	- Elastic IP
	- EC2
- Consistent performance:
	- Internal AWS network is used.
	- Intelligent routing and fast failover
	- No issues with client cache since IP does not change
- Health Checks:
	- Global Accelerator performs health check of our applications
	- Great for making application Global
	- Great for disaster recovery (Health checks)
- Security: 
	- AWS shield integration = DDoS protection inbuilt
	- only 2 IPs need to be whitelisted

### Working
- User tries to connect to the application
- User is provided with 2 anycast IPs (in case one ip has issues)
- These IPs route the user to AWS's internal network
- And finally the request reaches the server

### Usecase:
- Users are spread in different regions
- Need low latency
- Static IPs required
Dont use when:
- Cost matters a lot
- Users are in one region -> No big improvement
- Cloudfront already solves our problem

### Cloudfront vs Global Accelerator

| Cloudfront                     | Global Accelerator                |
| ------------------------------ | --------------------------------- |
| It is a CDN (Used for caching) | It is for faster network routing  |
| Do i need to hit the server?   | What is the fastest way to server |
| No static IP                   | Has static IP                     |
| HTTP/HTTPS                     | TCP/UDP                           |
| Images / videos                | gaming, VoIP, real-time apps      |
